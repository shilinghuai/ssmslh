1.1 产品简介

统一开发平台是一个基于SpringBoot、SpringCloud的微服务架构的基础开发平台，通过其提供的基础框架、内建的多渠道支持、丰富的基础组件、强大的开发工具及运维工具，让用户可以快速构建微服务架构的应用。同时统一开发平台也在平台级做了适配，让用户可以快速构建模块化的单体应用。

统一开发平台提供了前、后台一站式开发支撑，通过DevOps工具来辅助开发和运维以达到提高开发效率、降低开发门槛的目的，通过服务治理组件来解决分布式系统下的技术难点。

统一前端平台和统一移动平台是基于VUE的PC端和移动端开发框架，前后端完全分离，和后台通过RESTful通讯。

分布式服务计算平台是平台封装了开箱即用的技术组件，通过提供成熟、稳定、易用的技术组件方便开发人员快速开发应用系统。统一开发平台对关系型数据库、NoSQL数据库、Redis、ES等基础设施提供封装易用的API。

统一服务平台通过网关（API Gateway）对外提供基于RESTful的服务，网关统一实现了后台微服务需要但与业务逻辑无关的一些功能，如：安全认证，微服务架构下的单点登陆功能，通过网关统一认证；访问控制，微服务架构下的控制点、菜单等权限访问控制。作为API网关同时还需要具备动态路由、负载均衡的能力以便实现更加轻量的反向路由后台微服务。

微服务层是具体的业务逻辑实现层，并通过网关提供RESTful接口，分为聚合微服务和原子微服务。原子微服务提供“原子”服务是与会话无关的，聚合微服务是对原子服务进行组合、编排后提供的“复合”服务，并在原子服务基础上加入了会话状态和业务逻辑校验等。

基础组件层提供微服务开发所需要的技术支撑，如缓存组件、消息组件、分布式事务组件等。

1.1 产品价值

随着信息技术的不断发展，企业对应用系统的各个方面的要求越来越高，单体应用中普遍存在的一些问题也越来越受到关注，譬如系统的性能、开发效率、开发质量、开发成本、可复用性、可扩展性等。统一开发平台正是为了解决这些问题而诞生：

n 快速迭代能力，快速响应业务需求；

n 高性能、水平扩展能力，可以按需来水平扩展应用的吐吞量；

n 简单运维能力，通过管控工具可以极大提高分布式系统的运维能力；

n 去IOE能力，降底服务器采购成本；

n 组件积累、复用能力，通过基础组件的积累复用可以有降低开发成本；

n IDE辅助开发能力，通过IDE来完成代码自动生成、检查保证用户的开发效率和代码质量。

2.1.4 分布式缓存

基于注解实现的缓存开发，保证开发人员对缓存开发无感知，可以在不改变应用代码的情况下更换Redis、Memcached等其它缓存数据库。

2.1.5 分布式事务

应用系统在拆分过程中避免不了会出现分布式事务，统一开发平台封装了“PCR”、“TCC”、“消息事务最终一致性”三种事务开发方式，针对不同的应用场景可以选择不同分布式事务组件。

2.1.6 分布式消息中心

消息中心作为统一开发平台中的消息服务组件，负责平台中所有的消息处理。主要的功能如下：

n 实时消息发送，在调用实时消息发送API时，使用非阻塞方式处理，最大程度保证应用不受消息中心影响；

n 消息订阅，支持按用户、角色、机构、岗位来订阅消息类型，可以订阅消息投放方式，如短信、站内信、邮件等；

n 消息模板，同一消息类型根据投放方式可以有不同的消息模板，如邮件会有标题、图片、附件等，短信是纯文本内容；

n 投放扩展，缺省实现短信、邮件、站内信的方式，通过投放扩展出更多实现；

n 投放时间，订阅类的消息支持设置投放时间范围设置，避免夜间投放短信打扰用户；

n 消息队列，所有的消息进入消息队列，通过队列削峰填谷；

n 消息级别，根据消息类型可以设置消息级别，级别高的优先发送，实时消息默认级别高于订阅消息；

n 异常处理，支持发送失败的消息重复发送，重复指定次数发送失败的消息可以人工处理。

2.1.7 分布式锁

在单机环境中，Java中其实提供了很多并发处理相关的API，但是这些API在分布式场景中就无能为力了。也就是说单纯的Java Api并不能提供分布式锁的能力。统一开发平台基于Redis实现的分布式锁，开发简单、性能高效。

2.1.8 分布式文件系统

统一开发平台提供分布式环境下统一的文件处理：上传、下载、删除和文件基本信息查看等功能。支持FastDFS和NFS两种文件存储，并提供一致性编程。

2.1.9 全局序列号

全局序列号组件解决分布式架构下的全局序列号(有续)唯一性问题：

n 支持基于Oracle,Mysql,Db2,Sql Server2012的sequence生成；

n 支持非关系型数据库：Redis的自增序列；

n 开发者无需关注底层实现，只需配置使用何种场景下的自增序列；

n 根据配置模版，可生成带有业务含义的序列；

n 封装SDK，便于其他工程直接引入;

n 应用端序列缓存

2.1.10 分布式定时调度

基于xxl-job封装的分布式定时调度组件，核心目标是：开发迅速、学习简单、轻量级、易扩展。主要特性如下：

n 简单：支持通过Web页面对任务进行CRUD操作，操作简单，一分钟上手；

n 动态：支持动态修改任务状态、暂停/恢复任务，以及终止运行中任务，即时生效；

n 调度中心HA（中心式）：调度采用中心式设计，“调度中心”基于集群Quartz实现并支持集群部署，可保证调度中心HA；

n 执行器HA（分布式）：任务分布式执行，任务"执行器"支持集群部署，可保证任务执行HA；

n 注册中心: 执行器会周期性自动注册任务, 调度中心将会自动发现注册的任务并触发执行。同时，也支持手动录入执行器地址；

n 弹性扩容缩容：一旦有新执行器机器上线或者下线，下次调度时将会重新分配任务；

n 路由策略：执行器集群部署时提供丰富的路由策略，包括：第一个、最后一个、轮询、随机、一致性HASH、最不经常使用、最近最久未使用、故障转移、忙碌转移等；

n 故障转移：任务路由策略选择"故障转移"情况下，如果执行器集群中某一台机器故障，将会自动Failover切换到一台正常的执行器发送调度请求。

n 失败处理策略；调度失败时的处理策略，策略包括：失败告警、失败重试；

n 失败重试：调度中心调度失败且启用"调度失败重试"策略时，将会自动重试一次；执行器执行失败且启用"执行失败重试"策略，或回调失败重试状态时，也将会自动重试一次；

n 阻塞处理策略：调度过于密集执行器来不及处理时的处理策略，策略包括：单机串行（默认）、丢弃后续调度、覆盖之前调度；

n 任务超时控制：支持设置任务超时时间，任务运行超时的情况下，将会主动中断任务；

n 分片广播任务：执行器集群部署时，任务路由策略选择"分片广播"情况下，一次任务调度将会广播触发集群中所有执行器执行一次任务，可根据分片参数开发分片任务；

n 动态分片：分片广播任务以执行器为维度进行分片，支持动态扩容执行器集群从而动态增加分片数量，协同进行业务处理；在进行大数据量业务操作时可显著提升任务处理能力和速度。

n 事件触发：除了"Cron方式"和"任务依赖方式"触发任务执行之外，支持基于事件的触发任务方式。调度中心提供触发任务单次执行的API服务，可根据业务事件灵活触发。

2.1.11 读写分离、分库分表

基于MyCat实现的读写分离、分库分表组件，对应用开发透明无感，自动将查询类SQL路由到读库，并提供应用可干预的读写分离扩展API。根据分片规则（可自定义）将大表数据水平切分存储到不同的数据库，小表自动广播到各数据库。

2.2 微服务管控平台

2.2.1 首页仪表盘

为操作员提供一个统一的视图展示当前操作员所属集群概要信息。

2.2.2 应用登记

提供微服务应用的新增、修改、删除上传和自动构建等操作，操作员可通过此功能实现微服务应用的登记，并与jenkins集成进行自动构建操作。

2.2.3 应用部署

提供微服务应用的多实例部署、灰度发布、部署监控和部署历史查询等功能，操作员可以通过此功能快速的部署多个微服务实例。

2.2.4 应用管理

提供微服务应用的多个实例的管理，操作员可以进行实例的快速启停、实例的运行状态查看、断路器查看、并设置实例的权重进行负载均衡与灰度发布，还可以查看到每个实例的控制台日志。

2.2.5 弹性伸缩

提供微服务应用的自动和手动弹性伸缩功能，操作员可通过自定义配置弹性伸缩指标来达到动态扩容微服务应用的目的，也可以通过手动扩容操作来对微服务应用进行扩容。

2.2.6 服务限流

提供微服务应用的限流配置，操作员可通过IP地址、登录用户、请求URI、每秒请求次数、并发数等策略对微服务进行限流操作，同时可以开启或关闭相应的限流策略。

2.2.7 服务降级

提供微服务应用的降级配置，操作员可对每一个微服务应用里的任意api进行多种降级策略的配置，如：屏蔽降级、返回null、返回指定异常、执行mock等。

2.2.8 中间件部署

提供微服务管控平台常用中间件的部署功能，操作员可以对中间件部署情况进行查看，并可以启动和停止中间件。

2.2.9 应用参数配置

提供基于Spring cloud config的配置中间管理，支持yml和键值对的查询方式，对配置进行新增、修改、删除，并可动态下发配置。

2.2.10 网关过滤器启停

提供微服务应用网关已配置过滤器的启停操作，操作员通过对过滤器的启停达到通过预设的网关过滤器来实时控制网关流量的效果。

2.2.11 预警规则配置

提供微服务应用和主机的预警规则配置，操作员可以对主机的cpu,内存，硬盘使用率进行预警；还可以对微服务的响应时间进行预警；预警方式可选择email、微信、短信方式。

2.2.12 主机监控

提供主机的CPU、内存、磁盘的使用率等指标进行监控。

2.2.13 中间件监控

提供微服务管控平台基础中间件设施，如：Redis、MySQL、RabbitMQ等。

2.2.14 应用监控-服务APM监控

提供微服务应用的APM监控，操作员可以查看微服务应用的平均响应时间，总交易数，错误数，并且可以进行排序。

2.2.15 应用监控-API监控

提供微服务应用的API访问概况和访问明细的监控。

2.2.16 应用监控-服务监控

提供微服务应用的内存、线程、Web请求和响应的监控。

2.2.17 应用监控-链路监控

提供微服务应用的调用链监控。

2.2.18 日志分析

基于ELK的日志查询和分析系统

2.2.19 预警日志

提供微服务应用和主机的预警日志的查询和展示。

2.2.20 集群管理

提供集群的增删改查功能。

2.2.21 主机管理

提供主机的增删改查功能，同时提供了网页上运行主机命令的功能。

2.4 前端开发平台

2.4.1 多渠道开发支持

同一套前端开发框架，支持4个渠道开发：

n PC端：内管系统，如客户关系管理、绩效考核系统、管理驾驶舱、运营风险监控等，互联网系统，如网银、门户系统、积分商城等。

n 手机端：支持移动应用如手机银行、掌上驾驶舱、掌上营销、移动营销、移动信贷等。

n PAD端：支持移动展业、厅堂营销等。

n 自助设备：支持超级柜台、智能网点等。

2.4.2 完备的技术组件、业务组件

前端业务功能开发组件化，针对不同渠道设备提供了不同的技术组件与业务组件。

n PC：提供41个基础组件，7个业务组件积累、沉淀。

n 手机：提供34个基础组件，6个业务组件。

n PAD与自助设备：提供34个基础组件，7个业务组件。

2.4.3 业务模板快速开发

根据以往业务功能需求抽象出不同设备的业务模板，开发时直接使用，提升开发效率。其中PC端16套模板，手机端5套模板，PAD端5套模板。

2.4.4 前后端分离架构

前后端分离的架构，实现了架构的解耦。前端只关注页面表现，速度流畅，兼容性，用户体验。前端可完全独立开发，独立部署，服务化。前端通过Ajax请求调用后端Restful Api接口，以JSON格式报文进行交互。并且前端可通过Mock拦截Ajax请求，模拟后端接口返回报文，使得前端不依赖于后端开发，实现开发与联调接口上线使用同一套代码。

2.4.5 多语言支持

支持多语言配置，默认使用中文，技术组件内置三种语言包：简体中文（zh-CN）、繁体中文（zh-TW）、英语（en）。多语言具体实现方案是通过统一模板文件，前端根据相应的语言映射包去做文案的替换。同时提供将业务功能快速转换为模板文件及默认中文语言包的转换工具。针对要求实现多语言的系统，仅需翻译语言包放到指定目录，将平台多语言开关打开即可。只开发一套代码即实现多语言支持。
